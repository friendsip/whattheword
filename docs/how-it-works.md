# How "What the Word?!" Works

"What the Word?!" is a multiplayer word-guessing party game. Teams of two take turns describing words to their partner without saying the word itself. The team with the most points wins.

This document describes the full chain of events from game creation through gameplay to completion.

## Game Creation

1. The **host** opens the app at `/` and fills out a form with:
   - Number of teams (2-10)
   - Team names
   - Number of rounds (1-10)
   - Round duration (30-180 seconds)
   - Word category/genre (General, Christmas, Movies, Friends, etc., or Custom)

2. A `POST /api/games` request is made. The server:
   - Generates a unique 6-character alphanumeric game code
   - Fetches 20 words from the Anthropic Claude Haiku API (falls back to a hardcoded word list on failure)
   - Clamps round duration to 30-180s and total rounds to 1-10
   - Creates the initial game state object
   - Stores it in Vercel KV (Redis) with a 1-hour TTL

3. The host sees the game code and shareable links for each team (e.g. `/game/ABC123/team/0`). Share buttons for WhatsApp, SMS, and clipboard are provided.

---

## Teams Joining

There is no explicit "join" step. The host defines all teams at game creation. Team members simply open the link shared with them, which loads the Player component. The player is identified by the URL parameters (`gameId` and `teamIndex`). Once the page loads, the client begins polling the server every 1 second for game state.

## Round Lifecycle

The game uses a **timestamp-based state machine** because serverless functions cannot hold timers between invocations. Every API call runs `updateGameState(game)`, which checks timestamps against the current time and transitions state accordingly.

### Phase 1: Waiting for Ready

- Each team sees a "We're Ready" button
- Tapping it calls `POST /api/games/{gameId}/team/{teamIndex}/ready`, which sets `team.ready = true`
- Once **all** teams are marked ready, the server calls `startCountdown(game)`, setting `countdownEndsAt = now + 5000`

### Phase 2: Countdown (5 seconds)

- Clients display a countdown timer derived from the `countdownEndsAt` timestamp
- On the next API call after the countdown expires, `updateGameState()` detects `now >= countdownEndsAt` and transitions the game:
  - Sets `gameStarted = true` and `roundActive = true`
  - Sets `roundEndsAt = now + (roundDuration * 1000)`
  - Clears `countdownEndsAt`
  - Resets all teams' `ready` flags to `false`

### Phase 3: Round Active

- Teams see the current word and two buttons: **"Got it!"** and **"Skip"**
- Tapping either calls `POST /api/games/{gameId}/team/{teamIndex}/guess` with `{ correct: true/false }`
  - **Got it!** (`correct: true`): Team score increments by 1, word advances
  - **Skip** (`correct: false`): Word advances, no score change
- Words cycle using modular arithmetic: `(currentWordIndex + 1) % words.length`
- Each team tracks its own `currentWordIndex` independently, so teams progress through words at their own pace
- A timer counts down from `roundDuration` to 0, derived from the `roundEndsAt` timestamp
- When the server detects `now >= roundEndsAt`, it sets `roundActive = false` and `waitingForHost = true`

### Phase 4: Waiting for Host

- Teams see a "Waiting for host" message
- The host sees the scoreboard and a "Start Round X" button (or "End Game & See Results" for the final round)
- Pressing it calls `POST /api/games/{gameId}/nextround`, which:
  - Increments `currentRound`
  - Resets `waitingForHost` to `false`
  - Resets all teams' `ready` flags and `currentWordIndex` values to `0`
  - Fetches **fresh words** from the Anthropic API for the new round
- The cycle returns to [Phase 1](#phase-1-waiting-for-ready)

## Word Generation and Cycling

### Generation

Words are generated by calling the Anthropic Claude Haiku API via `getWords(genre)` in `lib/words.js`. This happens at two points:
- On initial game creation
- At the start of every new round

The function selects a prompt based on the genre (from a hardcoded `genrePrompts` map or a dynamic prompt for custom genres), requests 20 words, and parses the response (one word per line, uppercased, trimmed).

If the API call fails for any reason, a hardcoded fallback list of 10 common nouns is used instead.

### Cycling

During a round, each team has a `currentWordIndex` that advances on every guess (correct or skip):

```
team.currentWordIndex = (team.currentWordIndex + 1) % game.words.length
```

This wraps around, so after the last word the team cycles back to the first. Teams track their word position independently.

---

## Scoring

- Each team starts at `score = 0`
- A correct guess ("Got it!") adds 1 point
- A skip adds nothing; there are no penalties
- Scores persist across rounds and only reset on game restart
- The host's scoreboard displays all team scores in real time (updated every 1-second poll)

### Winner Determination

At game end, teams are sorted by score. If the top two teams are tied, the result is declared a tie. Otherwise the highest-scoring team wins.

---

## Game End

1. After the final round ends, the host sees "End Game & See Results"
2. Calling `/nextround` when `currentRound` would exceed `totalRounds` sets `gameComplete = true` and returns the winner
3. **Host view**: Displays final rankings for all teams and a "Play Again?" option with genre selector
4. **Player view**: Displays the team's final score and a message to check the host's screen for full results

### Play Again

The host can press "Play Again?" to call `POST /api/games/{gameId}/restart` with a new genre. This resets all state (scores, rounds, ready flags, words) while keeping the same game code and team URLs, so players don't need new links.

### Cleanup

Games are stored in Redis with a 1-hour TTL and are automatically deleted after expiry.

## Architecture

### State Machine

The game state is driven by a set of boolean flags and timestamps:

| State | Key Flags | Trigger to Next State |
|---|---|---|
| Setup / Waiting for Ready | `roundActive=false`, `countdownEndsAt=null` | All teams call `/ready` |
| Countdown | `countdownEndsAt` is set | 5 seconds elapse |
| Round Active | `roundActive=true`, `roundEndsAt` is set | `roundDuration` elapses |
| Waiting for Host | `waitingForHost=true` | Host calls `/nextround` |
| Game Complete | `gameComplete=true` | Host calls `/restart` (optional) |

### Polling

There are no WebSockets or server-push mechanisms. Both the host and player views poll their respective API endpoints every 1 second using `setInterval`:

- **Host**: `GET /api/games/{gameId}` (returns all teams, scores, round info)
- **Player**: `GET /api/games/{gameId}/team/{teamIndex}` (returns team-specific view)

Each poll response updates the component's local state, triggering a React re-render.

### Timestamp-Based Timing

Since Vercel serverless functions are stateless and cannot hold timers, all timing is based on stored timestamps:

- `countdownEndsAt`: When the 5-second countdown expires
- `roundEndsAt`: When the current round expires

Every API handler calls `updateGameState(game)` before processing the request. This function compares `Date.now()` against these timestamps and transitions state if a deadline has passed. Clients compute display timers (countdown seconds, time remaining) from these same timestamps.

### Storage

The entire game is a single JSON object stored in Vercel KV (Redis) under the key `game:{gameId}`. Every API call reads the full object, potentially mutates it, and writes it back. The KV wrapper in `lib/kv.js` provides `getGame()`, `setGame()`, and `deleteGame()`.

There is no authentication. Access is controlled by knowledge of the game code and team index in the URL.
